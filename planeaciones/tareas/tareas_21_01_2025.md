## ÉPICA 1: OPTIMIZACIÓN DEL FORMULARIO DE COTIZACIÓN MOBILE

### 🎫 Historia de Usuario 1.1: Borrado de Imágenes en Mobile
**Como** cliente usando dispositivo móvil  
**Quiero** poder eliminar las imágenes de referencia que subo desde mi celular  
**Para** corregir errores o cambiar imágenes sin tener que recargar el formulario

**Prioridad:** CRÍTICA  
**Story Points:** 5  
**Sprint:** Sprint 1  
**Assigned to:** Cami

#### Criterios de Aceptación
```gherkin
DADO que soy un usuario en dispositivo móvil
CUANDO subo una imagen de referencia al formulario de cotización
ENTONCES debe aparecer un botón "X" visible para eliminar la imagen

DADO que he subido múltiples imágenes
CUANDO hago clic en el botón "X" de una imagen específica
ENTONCES esa imagen debe eliminarse del formulario
Y las demás imágenes deben permanecer intactas
Y el contador de imágenes debe actualizarse

DADO que elimino una imagen

ENTONCES debe mostrarse feedback visual de confirmación
```

- [ ] No hay memory leaks en el proceso de eliminación

## ÉPICA 2: REDISEÑO DEL SISTEMA DE LOCACIONES

### 🎫 Historia de Usuario 2.1: Búsqueda Dinámica de Ubicación Estilo Google Maps
**Como** cliente queriendo encontrar un tatuador  
**Quiero** buscar mi ubicación mediante un solo campo de búsqueda autocompletable  
**Para** simplificar el proceso y encontrar rápidamente la ciudad donde quiero tatuar

**Prioridad:** ALTA  
**Story Points:** 13  
**Sprint:** Sprint 1-2  
**Assigned to:** Cami (Frontend), Dani (Backend/Integration)

#### Criterios de Aceptación
```gherkin
DADO que estoy en el formulario de cotización
CUANDO veo el campo de ubicación
ENTONCES debe ser un único campo de búsqueda con placeholder "¿Dónde quieres tatuarte?"

DADO que empiezo a escribir en el campo de ubicación
CUANDO escribo al menos 2 caracteres
ENTONCES debe aparecer un dropdown con sugerencias de ciudades
Y cada sugerencia debe mostrar formato "Ciudad, Departamento/País"
EJEMPLO: "Cali, Colombia" o "La Unión, Nariño"

DADO que veo la lista de sugerencias
CUANDO selecciono una ciudad
ENTONCES el campo debe autocompletarse con ese valor
Y la búsqueda de tatuadores debe filtrarse automáticamente
Y el dropdown debe cerrarse

DADO que escribo un texto que no coincide
CUANDO no hay resultados
ENTONCES debe mostrarse mensaje "No se encontraron ciudades"
Y opción para "Buscar de todas formas" o contacto manual

DADO que he seleccionado una ciudad previamente
CUANDO hago clic en el campo nuevamente
ENTONCES puedo modificar mi selección
Y ver el historial de búsquedas recientes (opcional)
```

#### Tareas
- [ ] Investigar y seleccionar API de geocoding (Google Places, Mapbox u otro) (2h)
- [ ] Configurar API en backend con endpoint de búsqueda y rate limiting (3h)
- [ ] Implementar componente de autocompletado con debouncing y navegación por teclado (4h)
- [ ] Integrar con formulario existente reemplazando campos múltiples (2h)
- [ ] Testing y optimización de performance (2h)

#### Definición de Done
- [ ] Único campo de búsqueda funcional
- [ ] Autocompletado responde en < 500ms
- [ ] Formato "Ciudad, Departamento/País" implementado
- [ ] Funciona correctamente en mobile y desktop
- [ ] API de geocoding integrada y funcionando
- [ ] Rate limiting configurado
- [ ] Error handling robusto
- [ ] Tests unitarios y de integración
- [ ] Code review aprobado
- [ ] Documentación API actualizada

#### Dependencias
- API de geocoding (Google Places, Mapbox o similar)
- Posible necesidad de actualizar base de datos de ubicaciones

#### Riesgos
- **MEDIO:** Costos de API si el tráfico es alto
- **MEDIO:** Performance si no se implementa debouncing correcto
- **BAJO:** Limitaciones de cobertura en ciudades pequeñas

## ÉPICA 3: REORGANIZACIÓN DEL FLUJO DEL FORMULARIO (UX)

### 🎫 Historia de Usuario 3.1: Reordenamiento del Formulario (Contacto al Final)
**Como** diseñador de UX de la plataforma  
**Quiero** reorganizar el orden de los pasos del formulario  
**Para** reducir la fricción inicial y aumentar la tasa de completación

**Prioridad:** ALTA  
**Story Points:** 5  
**Sprint:** Sprint 1  
**Assigned to:** Cami

#### Criterios de Aceptación
```gherkin
DADO que un cliente entra al formulario de cotización
CUANDO ve el primer paso
ENTONCES debe ver primero la selección de UBICACIÓN y ESTILO
Y NO debe ver campos de datos personales (nombre, email, teléfono)

DADO que el cliente avanza en el formulario
CUANDO completa ubicación, estilo y selección de tatuador
ENTONCES debe llegar al paso de datos de contacto
Y este debe ser el ÚLTIMO paso antes de confirmar

DADO que el cliente está en el último paso
CUANDO ve el formulario de contacto
ENTONCES debe ver campos: nombre, email, teléfono, descripción
Y debe ver resumen de su cotización (ubicación, estilo, tatuador)
Y botón final "Enviar cotización"

DADO que el cliente abandona el formulario antes del último paso
CUANDO no ha proporcionado datos de contacto
ENTONCES NO se debe guardar información personal (GDPR compliance)
Y puede retomar después si usamos cookies/session storage (opcional)
```

#### Tareas
- [ ] Mapear y documentar flujo actual vs flujo nuevo del formulario (1h)
- [ ] Reestructurar componentes moviendo contacto al final (3h)
- [ ] Actualizar lógica de guardado parcial con session storage (2h)
- [ ] Actualizar tests e2e para nuevo flujo (1h)

#### Definición de Done
- [ ] Datos de contacto movidos al último paso del formulario
- [ ] Ubicación y estilo son los primeros campos visibles
- [ ] Barra de progreso actualizada correctamente
- [ ] Navegación entre pasos funciona sin errores
- [ ] No se guardan datos personales hasta completar formulario
- [ ] Tests e2e pasando
- [ ] Testing manual en diferentes escenarios
- [ ] Analytics configurado para medir abandono por paso
- [ ] Deployed en staging para pruebas

#### Dependencias
- Puede implementarse en paralelo con otras historias de formulario

#### Riesgos
- **MEDIO:** Posible impacto en tasa de conversión (requiere A/B testing posterior)
- **BAJO:** Usuarios acostumbrados al flujo anterior pueden confundirse inicialmente

## ÉPICA 4: SISTEMA DE FILTROS Y BÚSQUEDA DE TATUADORES

### 🎫 Historia de Usuario 4.1: Filtros de Ubicación y Estilo como Primeros Criterios
**Como** cliente buscando un tatuador  
**Quiero** que la ubicación y el estilo sean los primeros filtros visibles  
**Para** ver solo tatuadores relevantes que puedan realizar mi tatuaje

**Prioridad:** ALTA  
**Story Points:** 13  
**Sprint:** Sprint 2  
**Assigned to:** Cami (Frontend), Dani (Backend)

#### Criterios de Aceptación
```gherkin
DADO que un cliente inicia la búsqueda de tatuador
CUANDO accede al formulario/página de búsqueda
ENTONCES debe ver dos campos principales ANTES de cualquier otra información:
  1. Campo de búsqueda de ubicación (autocompletado)
  2. Selector de estilo de tatuaje

DADO que selecciono ubicación y estilo
CUANDO completo ambos campos
ENTONCES la lista de tatuadores debe filtrarse automáticamente

  - Trabajen en la ciudad seleccionada
  - Realicen el estilo de tatuaje seleccionado

ENTONCES debe mostrarse mensaje claro:

Y mostrarse loading indicator durante la búsqueda
```

- [ ] Performance < 1s para consultas típicas
- [ ] Estados vacíos implementados con UX clara
- [ ] Resultados se actualizan en tiempo real al cambiar filtros
- [ ] Responsive en mobile y desktop
- [ ] Tests unitarios y de integración
- [ ] Testing con datasets realistas

## 🚨 RIESGOS Y MITIGACIONES

### RIESGOS TÉCNICOS

#### 1. Performance de Búsqueda con Filtros Múltiples
**Riesgo:** ALTO  
**Impacto:** Búsquedas lentas (> 2s) pueden frustrar usuarios  
**Mitigación:**
- Implementar índices en BD desde el inicio
- Usar caching (Redis) para búsquedas frecuentes
- Paginación y lazy loading
- Testing de carga antes de lanzar

#### 2. Doble Reserva en Sistema de Calendario
**Riesgo:** ALTO  
**Impacto:** Dos clientes agendando mismo espacio causa conflictos  
**Mitigación:**
- Usar transacciones DB con nivel de aislamiento adecuado
- Implementar row-level locking
- Mostrar "Este espacio ya no está disponible" si conflicto
- Tests de concurrencia

#### 3. Integración con APIs de Terceros (Geocoding, Calendar)
**Riesgo:** MEDIO  
**Impacto:** Dependencia de servicios externos  
**Mitigación:**
- Implementar rate limiting
- Caching de respuestas comunes
- Fallback a servicios alternativos
- Manejo robusto de errores y timeouts
- Monitoring de uptime de APIs

#### 4. Manejo de Zonas Horarias en Calendario
**Riesgo:** MEDIO  
**Impacto:** Confusión en horarios entre tatuador y cliente  
**Mitigación:**
- Guardar todos los timestamps en UTC
- Mostrar siempre zona horaria explícitamente
- Usar librería robusta (moment-timezone, day.js)
- Testing con diferentes zonas horarias

### RIESGOS DE PRODUCTO

#### 5. Cambio en Flujo del Formulario Puede Afectar Conversión
**Riesgo:** MEDIO  
**Impacto:** Tasa de conversión puede bajar temporalmente  
**Mitigación:**
- Implementar analytics detallado por paso
- A/B testing antes de roll out completo
- Revertir fácilmente si métricas empeoran
- Monitorear tasas de abandono post-lanzamiento

#### 6. Tatuadores No Configuren Sus Perfiles Correctamente
**Riesgo:** MEDIO  
**Impacto:** Búsquedas no encuentran tatuadores relevantes  
**Mitigación:**
- Onboarding guiado para tatuadores nuevos
- Validaciones y campos requeridos
- Mensajes claros de por qué configurar cada sección
- Revisar perfiles manualmente al inicio

#### 7. Sistema de Calendario Demasiado Complejo para Algunos Tatuadores
**Riesgo:** MEDIO  
**Impacto:** Tatuadores no adoptan la funcionalidad  
**Mitigación:**
- Hacer calendario opcional (toggle on/off)
- Proveer defaults sensibles
- Videos tutoriales cortos
- Soporte personalizado a early adopters

### RIESGOS OPERACIONALES

#### 8. Costos de APIs de Terceros Pueden Escalar
**Riesgo:** MEDIO  
**Impacto:** Costos operativos altos con crecimiento  
**Mitigación:**
- Negociar planes con volumen desde el inicio
- Implementar caching agresivo
- Monitorear uso mensualmente
- Considerar alternativas open source

#### 9. Emails de Recordatorios van a Spam
**Riesgo:** MEDIO  
**Impacto:** Clientes no ven recordatorios, inasistencia continúa  
**Mitigación:**
- Configurar SPF, DKIM, DMARC correctamente
- Usar servicio reputado (SendGrid, AWS SES)
- Permitir clientes agregar a lista blanca
- Monitorear tasas de entrega

## 🎓 GLOSARIO

- **DoD (Definition of Done):** Criterios que debe cumplir una historia para considerarse completa
- **Story Points:** Unidad de estimación de complejidad/esfuerzo (no horas)
- **Sprint:** Iteración de 1-4 semanas (en este caso, 2 semanas)
- **Épica:** Conjunto de historias relacionadas con un objetivo común
- **Velocity:** Story points completados por sprint (mide capacidad del equipo)
- **Backlog:** Lista priorizada de trabajo pendiente
- **Spike:** Investigación técnica time-boxed
- **Tech Debt:** Deuda técnica acumulada por decisiones rápidas
